# Tessera Invoice Inbox

Upload an invoice or receipt (PDF or image) and get back **structured, review-ready
JSON** — vendor, dates, currency, totals, and line items — each field tagged with a
green / amber / red confidence flag so a human knows exactly what to double-check
before exporting to CSV.

This is a portfolio demo for a fictional B2B SaaS, "Tessera." It is a sibling to two
other demos (`docs-chat`, `tessera-ops-agent`) and deliberately showcases the
**native Anthropic API** (`@anthropic-ai/sdk`) — where the siblings use
OpenAI-compatible routes — including PDF/vision document input and tool-use-enforced
structured output.

> Built with [Claude Code](https://www.anthropic.com/claude-code).

---

## What it does

1. **Upload** a PDF / JPG / PNG (drag-and-drop or file pick, capped at 4 MB and 5
   pages) — or click one of three **preloaded samples** (no upload needed).
2. **Extract** to a Zod-validated schema via Claude Haiku 4.5 using **tool use**, so
   the API enforces the output *shape* rather than us hopefully parsing free text.
3. **Review** in an editable field table. Every field carries a confidence badge
   combining two independent signals (see below). Low-confidence fields are visually
   flagged for review; edits update state and re-run the deterministic checks live.
4. **Export CSV** of the (possibly corrected) header + line items.

### Dual confidence signal

Each field's badge is the **worst of two signals**:

- **Model confidence** — the model reports `high` / `medium` / `low` per top-level
  field, alongside the values, in the same tool call.
- **Deterministic validation** — computed in code, not by the model:
  - `arithmetic_mismatch` — line items don't sum to the subtotal/total (or
    subtotal + tax ≠ total) within a small tolerance.
  - `format_warning` — a date field doesn't parse to a real calendar date.

A confident-but-wrong model can't paint a field green if the numbers don't add up.

## Setup

Requires Node 20+.

```bash
npm install
cp .env.example .env.local      # then edit .env.local
```

Set your key in `.env.local`:

```
ANTHROPIC_API_KEY=sk-ant-...
```

Get a key at <https://console.anthropic.com/>. Without it, `/api/extract` returns a
friendly **503** and the UI shows a notice — it never crashes.

```bash
npm run dev        # http://localhost:3000
npm run build      # production build
npm run typecheck  # tsc --noEmit
```

### Model

Extraction uses **Claude Haiku 4.5** (`claude-haiku-4-5`) — the cheapest current
Claude tier ($1 / $5 per million input / output tokens) that supports **both** PDF
document input and image/vision input. Document extraction is a bounded, low-reasoning
task, so the fastest/cheapest capable model is the right call. Swapping models is a
one-line change in [`lib/config.ts`](lib/config.ts), or set `EXTRACTION_MODEL` in the
environment to override without a code change.

### Provider routing (native Anthropic, or OpenRouter)

The client construction is provider-agnostic — it's the **same native
`@anthropic-ai/sdk`** either way; only the base URL and auth differ:

- **`ANTHROPIC_API_KEY` set** → direct Anthropic (production default; unchanged).
- **else `OPENROUTER_API_KEY` set** → the SDK targets OpenRouter's
  Anthropic-Messages-compatible endpoint (its "Anthropic Skin") at
  `https://openrouter.ai/api` with Bearer auth (`authToken`). The same
  `messages.create` + strict-tool-use path serves both.

Anthropic wins when both are set, so production is never affected. When routing
through OpenRouter, the model id must be addressed the OpenRouter way — set
`EXTRACTION_MODEL=anthropic/claude-haiku-4.5` (OpenRouter's id for the same model;
the native alias is `claude-haiku-4-5`). Env vars, all in `.env.local`:

```
# Native Anthropic (production default)
ANTHROPIC_API_KEY=sk-ant-...

# OR route the same native SDK through OpenRouter (e.g. for the eval)
OPENROUTER_API_KEY=sk-or-...
EXTRACTION_MODEL=anthropic/claude-haiku-4.5
```

### Sample documents

Three invented ("Tessera-fictional") vendors live in `public/samples/`, generated by

- `northwind-services-invoice.pdf` — a services invoice that fully reconciles.

## Abuse guards

The public `/api/extract` endpoint is protected at the boundary
([`lib/guards.ts`](lib/guards.ts), ported from the `docs-chat` sibling):

- **Per-IP sliding window** — 6 requests / minute, with a sweep of stale entries
  once the map exceeds 1,000 keys (bounded memory, no background timer).
- **Global UTC-day cap** — 200 served requests / day as a kill-switch.
- **Env-gated Cloudflare Turnstile** — skipped entirely if `TURNSTILE_SECRET_KEY` is
  unset; enforced if set.
- Blocked requests (Turnstile 403 or rate-limit 429) **never** increment the daily
  counter, and 429/403 render as a friendly JSON notice.

These in-memory limits are per-process best-effort; the real hard backstop is a
provider-side spend cap on the Anthropic key.

## Evaluation (SROIE)

`npm run eval` runs the **real** extraction pipeline over 30 committed receipts from
the **ICDAR-2019 SROIE** dataset and scores field-level exact-match accuracy after
normalization:

- `vendor → company` (case + whitespace)
- `issueDate → date` (parsed to canonical `YYYY-MM-DD`)
- `total → total` (currency symbols / separators stripped, numeric compare)

It prints a per-doc table plus per-field and overall accuracy, and **exits 1 below
75% overall**. It needs a provider key at runtime — `ANTHROPIC_API_KEY`, or
`OPENROUTER_API_KEY` (+ `EXTRACTION_MODEL=anthropic/claude-haiku-4.5`) to run the same
native SDK through OpenRouter (see [Provider routing](#provider-routing-native-anthropic-or-openrouter)).
Without a key it prints a friendly message and exits 1 (it is designed to fail
gracefully, not to pass without a key).

**Measured results** (30 SROIE receipts, `claude-haiku-4.5`, 2026-07-17):

| Field | Exact-match accuracy |
|---|---|
| total | **93.3%** (28/30) |
| company | 70.0% (21/30) |
| date | 70.0% (21/30) |
| **overall** | **77.8%** (70/90) — gate ≥75% ✅ |

Measured through OpenRouter's Anthropic-Messages endpoint — the same native-SDK
strict-tool-use request as production, and the score matches the direct-Anthropic
measurement exactly, which is the point of the provider-agnostic design
(DECISIONS.md #8). Scores are strict *exact-match after normalization*: most
company/date misses are OCR-adjacent near-misses on low-quality scans, while the
field that drives money decisions (`total`) reads at 93%.

**Provenance:** the eval images + ground truth in `data/eval/` come from the
HuggingFace mirror `jsdnrs/ICDAR2019-SROIE` (CC-BY-4.0), which originates from the
ICDAR 2019 Robust Reading Challenge on Scanned Receipts OCR and Information
Extraction. Full source URLs, citation, and the verbatim license/NOTICE are in
[`data/eval/PROVENANCE.md`](data/eval/PROVENANCE.md).

> Why SROIE receipts to grade an invoice extractor? Receipts aren't invoices, but
> using a **third-party, independently authored** benchmark is far more honest than
> grading against invoices we wrote ourselves. See [`DECISIONS.md`](DECISIONS.md).

## Project layout

```

components/      UploadZone, ResultView, ConfidenceBadge (plain CSS modules)
```
